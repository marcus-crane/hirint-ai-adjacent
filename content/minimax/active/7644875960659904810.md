## 职位描述

岗位背景

我们在为大模型做「移动应用开发」方向的后训练(SFT / RL / Agent 评测)。模型能力的上限,本质上由训练数据和评测体系的质量决定 —— 尤其在 Android 这种"既要写代码、又要在真实设备上跑得起来"的领域,造数据和搭评测的工程难度并不比训模型低。

我们需要一位真正懂安卓的工程师,把数据合成、评测、模拟器自动化这条链路做扎实。

你将负责

1. 训练数据构造:为安卓方向(原生 / Flutter / RN / KMP 等等)设计任务维度与配方,产出可用于 SFT / RL 的高质量数据,包括单轮代码题、多轮交互任务、UI 自动化任务等;

2. 自动化 / Agent / LLM 评测体系:构造真实、全面、准确地评估模型在安卓领域能力的评测体系 —— 设计与维护 Android 代码 / App 行为的自动化评测,包括但不限于编译验证、单测、UI 比对、功能回归、Agent 端到端评测 —— 让"模型生成的代码或操作是否真的有效"可被自动打分;

3. 安卓设备和沙盒集群:建设并维护 高并发、高可用、多样化 的安卓设备集群与沙盒链路,覆盖不同 Android 版本 / 厂商 ROM / 屏幕形态 / 网络条件,保证评测与训练场景的真实性与稳定性;扩展 Agent 在设备上的能力边界。

## 任职要求

硬性要求

- 认可并深度拥抱 AI:这是岗位的前提 —— 本岗位的产出对象本身就是 AI,对 AI 抱观望或抵触态度的同学请绕道。要求把 AI 编程工具当成日常生产力的一部分,熟练使用 Claude Code / Cursor / Codex 等至少一款,持续关注前沿 Agent 框架与 harness(Claude Code、mini-swe-agent、OpenHands、SWE-agent 等),知道它们各自的设计取舍;了解 skill / plugin / sub-agent 等机制的原理与作用,知道在什么场景下该用哪种;

- 真正理解大模型与 Agent 的工作原理(这一条甚至比"会写安卓"更重要):能清晰区分 SFT 数据 / RL reward / 评测集的差别,理解 prompt 设计 / tool calling / 多轮对话 / context engineering 这些概念到底在解决什么问题。不要求训过模型,但概念上要扎实 —— 你来这里不是写 App 的,是给模型造能力的,概念错了方向就错了;

- 3 年以上安卓相关工程经验,Kotlin / Java 熟练,熟悉 AGP / Gradle / R8;

- 熟悉 ADB / AVD / Android Emulator / KVM;

- Python 必备;TypeScript / Node.js 至少能读能改;

- 熟悉 Docker / Linux 与基础后端工程能力。

强加分项

- 做过 训练数据合成 / Benchmark 设计;

- 在 Agent / MCP 生态里有过 深度参与(魔改过 harness、写过 MCP server、给开源 Agent 项目提过 PR 等);

- Appium / UIAutomator2 自动化测试经验;

- JNI / NDK / OLLVM 与 安卓加固 / 混淆(R8、资源 / 字符串加密、Native 下沉、自研壳)研究或实战经验;

- Flutter / React Native / KMP 跨端经验;

- 了解 K8s / QEMU / 虚拟化 等基础设施技术,有大规模设备集群或沙盒平台搭建经验;

- Go 工程经验。
