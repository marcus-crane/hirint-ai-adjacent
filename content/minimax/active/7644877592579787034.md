## 职位描述

岗位背景

我们在为大模型做「移动应用开发」方向的后训练(SFT / RL / Agent 评测)。iOS 是一个比 Android 更"封闭、更挑工具链"的方向 —— 只能在 Mac 上构建、只有 Apple 自家的模拟器、签名 / Profile / 设备授权一堆细节;但模型在这条路径上的能力短板也最明显,后训练的杠杆很大。

模型能力的上限,本质上由训练数据和评测体系的质量决定,而 iOS 这种"既要写 Swift / SwiftUI 也要在真实模拟器跑得起来"的领域,造数据和搭评测的工程难度并不比训模型低。

我们需要一位真正懂 iOS 的工程师,把数据合成、评测、Mac 模拟器自动化这条链路做扎实。

你将负责

1. 训练数据构造:为 iOS 方向(SwiftUI / UIKit 原生 / Flutter / RN / KMP / 偶尔 macOS)设计任务维度与配方,产出可用于 SFT / RL 的高质量数据,包括单轮代码题、多文件 App 题、UI 自动化任务、Mac / iOS 跨端题等;

2. 自动化 / Agent / LLM 评测体系:设计与维护 iOS 代码 / App 行为的自动化评测 —— 包括但不限于 xcodebuild 编译验证、XCTest 单测、UI 比对、模拟器内功能回归、Agent 端到端评测 —— 让"模型生成的代码或操作是否真的有效"可被自动打分;

3. iOS 设备和沙盒集群:建设并维护 高并发、高可用、多样化 的 macOS Runner / iOS Simulator 集群与沙盒链路(Apple Silicon 虚拟化、tart / Anka / 自建方案都行),覆盖不同 iOS 版本 / 设备形态,保证评测与训练场景的真实性与稳定性;扩展 Agent 在 iOS 设备上的能力边界 —— 截图、点屏、看 view hierarchy、装 IPA、跑测试。

## 任职要求

硬性要求

- 认可并深度拥抱 AI:这是岗位的前提 —— 本岗位的产出对象本身就是 AI,对 AI 抱观望或抵触态度的同学请绕道。要求把 AI 编程工具当成日常生产力的一部分,熟练使用 Claude Code / Cursor / Codex 等至少一款,持续关注前沿 Agent 框架与 harness(Claude Code、mini-swe-agent、OpenHands、SWE-agent 等),知道它们各自的设计取舍;了解 skill / plugin / sub-agent 等机制的原理与作用,知道在什么场景下该用哪种;

- 真正理解大模型与 Agent 的工作原理(这一条甚至比"会写 iOS"更重要):能清晰区分 SFT 数据 / RL reward / 评测集的差别,理解 prompt 设计 / tool calling / 多轮对话 / context engineering 这些概念到底在解决什么问题。不要求训过模型,但概念上要扎实 —— 你来这里不是写 App 的,是给模型造能力的,概念错了方向就错了;

- 大厂 3 年以上 iOS 工程经验,Swift / SwiftUI 熟练,Objective-C 能读;业务覆盖面要广 —— 做过 C 端 / B 端 App ,对 App 架构、性能优化、网络 / 缓存、数据持久化等有广度认知;

- 熟悉 Xcode / xcodebuild / xcrun / simctl,能在命令行里把构建、签名、模拟器、跑测试这一套搞定,不依赖 Xcode IDE 也能干活;

- Python 必备;TypeScript / Node.js 至少能读能改;

- 熟悉 Docker / Linux 与基础后端工程能力,熟悉 macOS 系统管理(launchd / 用户与权限 / 网络 / 虚拟化)。

- XCTest / XCUITest / idb / Appium-iOS 自动化测试经验;

- Apple Silicon 虚拟化(tart / Anka / Virtualization.framework)、macOS Runner 集群搭建经验;

- 签名 / Provisioning Profile / MDM / TestFlight / 企业分发 实战经验;

- Flutter / React Native / KMP 跨端经验(Hermes / JSC / JS Bridge 调试加分);

- 逆向 / 加固(Mach-O、class-dump、Frida、混淆壳)研究或实战经验;
